PAYROLL ATTENDANCE BACKEND - STEP 1 TO STEP 25 IMPLEMENTATION SUMMARY

Purpose
-------
This document summarizes the full backend performance, scalability, reliability, security, observability, and production-readiness work completed from Step 1 to Step 25.

The main goal was to make payroll generation and read-heavy APIs safer for large data volumes, especially for employee counts growing from thousands toward 1 lakh+ records, while keeping Supabase Free Tier safe during development.


STEP 1 - MANUAL SCHEDULER API MOVED TO BACKGROUND JOB
-----------------------------------------------------
Problem:
The manual scheduler API was processing payroll directly inside the HTTP request. This caused timeout, fetch failed, and stress test failures.

Business logic change:
Manual scheduler trigger no longer generates payroll inside the API request.

New behavior:
POST /api/scheduler/run

Now does:
1. Checks if a scheduler run is already PENDING or RUNNING.
2. Creates a SchedulerRun record.
3. Adds a BullMQ job.
4. Returns immediately with HTTP 202.

Response:
{
  "success": true,
  "message": "Payroll scheduler started",
  "jobId": "scheduler_run_id",
  "status": "PENDING"
}

New/updated APIs:
POST /api/scheduler/run
GET /api/scheduler/runs/:id
GET /api/scheduler/runs

New infrastructure:
BullMQ
Redis
Payroll scheduler worker

Files added/updated:
src/jobs/payrollScheduler.queue.ts
src/jobs/payrollScheduler.worker.ts
src/jobs/payrollScheduler.processor.ts
src/worker.ts
src/modules/scheduler/*

STEP 3 - BATCH EMPLOYEE PROCESSING
----------------------------------
Problem:
Scheduler was loading all active employees at once.

Change:
Scheduler now processes active employees in batches.

Default:
SCHEDULER_BATCH_SIZE=50 for Supabase Free Tier.

Production later:
SCHEDULER_BATCH_SIZE=500 after DB upgrade/testing.

Repository methods added:
countActiveEmployees()
getActiveEmployeesBatch()

Result:
Memory pressure reduced.
Scheduler can process large employee counts more safely.


STEP 4 - REMOVE N+1 QUERIES FROM SCHEDULER
------------------------------------------
Problem:
For each employee, scheduler queried salary history and latest payroll separately.

Change:
Scheduler now preloads data per batch:
1. First salary histories for all employees in batch.
2. Latest payrolls for all employees in batch.

Added methods:
getFirstSalaryHistories(employeeIds)
getLatestPayrolls(employeeIds)

Result:
Before:
100 employees = 100 salary queries + 100 latest payroll queries.

After:
100 employees = 1 salary query + 1 latest payroll query.


STEP 5 - OPTIMIZE EXISTING PAYROLL DUPLICATE CHECK
--------------------------------------------------
Problem:
Scheduler called findPayroll() inside every employee-period loop.

Change:
Existing payrolls are preloaded per batch into a Map.

Added methods:
getExistingPayrollsForBatch()
getPayrollPeriodKey()

Result:
Duplicate checks are done from memory instead of repeated DB queries.


STEP 6 - OPTIMIZE PAYROLL CALCULATION INTERNALS
-----------------------------------------------
Problem:
Salary calculation fetched work-hour/overtime settings inside the daily loop.

Change:
Work-hour settings are loaded once per payroll period.

Added:
OvertimeService.getSettingsForDateRange()
OvertimeService.resolveSettingFromList()

Also optimized repository selects:
Salary histories now select only needed fields.
Attendance now selects only needed fields.
Advances now select only needed fields.

Payslip optimization:
PayrollService.generate() now accepts:
createPayslip?: boolean

Scheduler passes:
createPayslip: false

Result:
No DB query inside each working date loop for work-hour settings.
Scheduler payroll generation skips synchronous payslip creation.

STEP 12 - BULK WRITE OPTIMIZATION
---------------------------------
Principle:
Keep financial payroll creation safe in transaction.
Batch only safe log/item/ledger writes.

Changes:
Ledger entries use createMany where safe.
Ledger creation can run inside payroll transaction.
SchedulerRunItem uses createMany per batch.
Scheduler progress updates once per batch, not per employee.

Important:
No huge transaction for 1 lakh employees.
Payroll financial writes remain transaction-per-payroll.

STEP 14 - PAYROLL IDEMPOTENCY PROTECTION
----------------------------------------
Problem:
Duplicate payroll can happen through double-click, retries, overlapping jobs, or timeouts.

Change:
Added Payroll.activePayrollKey.

Rule:
For active payroll:
activePayrollKey = employeeId_periodStart_periodEnd

For cancelled payroll:
activePayrollKey = null

Unique constraint:
@@unique([activePayrollKey])

Behavior:
Only one active non-cancelled payroll can exist for employee + period.
Cancelled payroll releases the key and allows regeneration.

Added helper:
buildActivePayrollKey()

Also handles Prisma P2002 unique constraint error.
